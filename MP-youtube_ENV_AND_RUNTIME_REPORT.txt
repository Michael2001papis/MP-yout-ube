1. REQUIRED ENVIRONMENT VARIABLES (Firebase + demo admin list)
--------------------------------------------------------------------------------

The app reads these via Vite (`import.meta.env`). All six below MUST be non-empty
strings at build/dev time or `firebaseReady` stays false and features stay off.

  VITE_FIREBASE_API_KEY
  VITE_FIREBASE_AUTH_DOMAIN
  VITE_FIREBASE_PROJECT_ID
  VITE_FIREBASE_STORAGE_BUCKET
  VITE_FIREBASE_MESSAGING_SENDER_ID
  VITE_FIREBASE_APP_ID

Optional (defaults to empty / no admins if unset):

  VITE_ADMIN_EMAILS          (comma-separated emails treated as admin in UI only)

Source of truth in repo: `.env.example` (template with placeholder text).

--------------------------------------------------------------------------------
3. IS THE FIREBASE WARNING A REAL PROJECT ISSUE?
--------------------------------------------------------------------------------

YES. It is emitted by `src/services/firebase.ts` when any required variable is
missing or not a string. This is expected until a valid `.env` exists.

Restart after env changes: YES. Vite reads env vars when the dev server starts
(`npm run dev`) and when running `npm run build`. After creating or editing
`.env`, stop the dev server and start it again (or restart the build).

--------------------------------------------------------------------------------
4. FIREBASE READINESS = BLOCKER FOR WHICH FEATURES?
--------------------------------------------------------------------------------

When `firebaseReady` is false:

  • Auth: not initialized — login/register/Google/password reset cannot work.
  • Firestore: `db` is null — services return empty arrays / null where guarded.
  • Storage: not available — uploads will not run.

UI that still loads (read-only / static):

  • Routing, layout, header, pages that do not require data (About, Legal text,
    404).
  • Home can render but video lists stay empty; Watch may show “not found” or
    errors depending on route.

This matches the design: backend features are blocked without configuration.

IMPORTANT — Do not copy `.env.example` to `.env` and leave placeholders like
YOUR_API_KEY as-is: those strings are non-empty, so the code would set
`firebaseReady` true and call `initializeApp()` with invalid config, which can
produce confusing Firebase runtime errors instead of the clear “missing env”
path. Create `.env` only when you can paste real values from the Firebase
console (Project settings → Your apps → SDK setup).

--------------------------------------------------------------------------------
6. EXACT NEXT STEPS TO RUN LOCALLY WITH FIREBASE
--------------------------------------------------------------------------------

  1. In Firebase Console, open your project (or create one).
  2. Project settings → General → Your apps → Web app → copy config values.
  3. In MP-yout-ube project root, create a new file named `.env` (not committed).
  4. Paste real values matching the keys in `.env.example` (same names).
  5. Optionally set `VITE_ADMIN_EMAILS` to your test admin email(s).
  6. Run: `npm run dev` (restart if it was already running).
  7. Open the printed local URL (default http://localhost:5173/).

No placeholder-only `.env` should be committed; real secrets stay local.
