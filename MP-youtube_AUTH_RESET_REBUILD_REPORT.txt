MP-yout-ube — Auth flow reset & rebuild report
==============================================
This documents a structural reset of the `/auth` experience (not a label-only tweak).

--------------------------------------------------------------------------------
1. WHAT WAS REMOVED FROM THE OLD FLOW
--------------------------------------------------------------------------------

- **Single monolithic `AuthPage.tsx`** that mixed member UI, a large Firebase configuration block inside the same
  card, and a full administrator access section (violet panel, shield, CTAs to `/admin-access` or `/admin`).
- **Administrator UI inside the member card** — no longer part of the sign-in surface.
- **`FirebaseConfigStatus` on `/auth`** — the long “system configuration” panel is no longer embedded in the primary
  member card (that component remains available for `/admin-access` where operators need fuller context).
- **Redundant explanatory paragraphs** above disabled controls inside the card (replaced by one subtle footer link
  in the card when env is incomplete).
- **Duplicated Zod schemas in the page file** — moved to `authSchemas.ts` for a single source of truth.

--------------------------------------------------------------------------------
2. WHAT WAS REBUILT
--------------------------------------------------------------------------------

- **`AuthPage.tsx`** — always centered **single column** (`max-w-md`): `MemberAuthCard` + page footer with
  “Administrator entry” → `/admin-access`.
- **`MemberAuthCard.tsx`** — dedicated member experience:
  - Headlines for sign-in, register, and forgot password.
  - Google button, email/password flows, validation, navigation after success.
  - Auth errors from `AuthContext` when Firebase is ready.
  - Disabled controls when `!firebaseReady` with `opacity` + `pointer-events` for intentional inactivity.
  - When disabled: **one short line** under the subtitle (README / `.env.example` / restart) — no side panel, no
    variable list on `/auth`.
- **`authSchemas.ts`** — `loginSchema`, `registerSchema`, `forgotSchema` and exported types.
- **`AuthSetupSupportPanel.tsx`** — **removed** (was the dashed “Environment setup” column); operators still get full
  detail on `/admin-access` via `FirebaseConfigStatus` when needed.

--------------------------------------------------------------------------------
3. WHAT WAS SEPARATED
--------------------------------------------------------------------------------

| Concern              | Where it lives now                                      |
|---------------------|----------------------------------------------------------|
| Member sign-in/up   | `MemberAuthCard.tsx` only                               |
| Firebase / env help | One-line hint in `MemberAuthCard` when `!firebaseReady`; full list on `/admin-access` only |
| Admin / operators   | Footer link → `/admin-access`; dashboard still `/admin` for role `admin` |
| Full operator doc   | `AdminEntryPage` + existing `FirebaseConfigStatus`      |

--------------------------------------------------------------------------------
4. HOW MEMBER AUTH NOW WORKS
--------------------------------------------------------------------------------

- User opens `/auth` (or `/auth?mode=register`).
- **Google** and **email** flows call the same `AuthContext` methods as before.
- After success, `navigate(fromPath)` uses router `state.from` when present, else `/`.
- **Forgot password** toggles inside the card without leaving the route (then returns to login mode).
- **No admin role checks** in the member card — admins sign in like any user; elevated UI remains on `/admin` and
  header when `role === 'admin'`.

--------------------------------------------------------------------------------
5. HOW FIREBASE STATUS IS NOW PRESENTED
--------------------------------------------------------------------------------

- **On `/auth`:** one compact line under the title/subtitle when Firebase is not ready (points to `.env.example`,
  `.env`, `npm run dev`, README). No `id="auth-setup-support"` block, no variable chip list, no dashed aside panel.
- **On `/admin-access`:** `FirebaseConfigStatus` still available for operators who need the full missing-key list.

--------------------------------------------------------------------------------
6. HOW ADMIN ACCESS IS NOW SEPARATED
--------------------------------------------------------------------------------

- **No** administrator panel, shield block, or “Open admin console” button on the member auth card.
- **Footer line** only: “Platform operator? **Administrator entry** · separate from member sign-in” → `/admin-access`.
- **`/admin-access`** unchanged in purpose: dedicated staff landing, optional `FirebaseConfigStatus`, sign-in CTA,
  non-admin messaging, Rules disclaimer.

--------------------------------------------------------------------------------
7. IS THE NEW FLOW CLEARER AND MORE STABLE?
--------------------------------------------------------------------------------

**Clearer:** Yes — one mental model per surface (members vs setup vs operators), fewer competing visual hierarchies.

**More stable:** Behavior is unchanged at the integration layer (`AuthContext`, Firebase init, guards). Stability gains
are structural: smaller files, explicit boundaries, easier to change member copy or support copy independently.

--------------------------------------------------------------------------------
8. WHAT STILL DEPENDS ON FIREBASE CONFIGURATION
--------------------------------------------------------------------------------

- All six `VITE_FIREBASE_*` variables must be non-empty for `firebaseReady` (`src/services/firebase.ts`).
- Until then: member controls stay disabled; short inline hint on `/auth`; **no** session or Firestore from the real app.
- **Google Sign-In** and **email auth** still require Firebase Console (providers, authorized domains) in addition to env.
- **Production security** still requires Firebase Security Rules (and/or backend) — unchanged by this UI reset.
