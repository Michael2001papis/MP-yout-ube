MP-yout-ube ‚Äî Admin access implementation report
==============================================
Generated for deployment and security review. This file describes how staff/admin access works in the current SPA.

1) How admin login is handled
---------------------------
- There is no separate ‚Äúadmin password‚Äù in the app. Staff use the same Firebase Authentication flows as everyone else (email/password or Google, per project settings).
- Which accounts are treated as administrators is determined at runtime from environment configuration: `VITE_ADMIN_EMAILS` (comma-separated). On first sign-in, `ensureUserDoc` in `src/context/AuthContext.tsx` assigns role `admin` when the signed-in email matches that list (case-insensitive).
- Passwords and API secrets must never be placed in `VITE_*` variables; they are bundled for the client. Configure credentials only through Firebase Console / secure server-side config.
- A public informational page at `/admin-access` (`AdminEntryPage`) explains staff-only access and links to `/auth?mode=login` with navigation state so returning users can be sent back toward `/admin` after login when appropriate.

2) How admin role is detected
-----------------------------
- After Firebase Auth resolves the user, the app loads (or creates) the Firestore document `users/{uid}` and reads `role` from it. New users get `admin` if their email is in `VITE_ADMIN_EMAILS`, otherwise `user`.
- If a `role` field already exists in the user document, that stored value is used (see `ensureUserDoc`). Promoting an existing account may require updating Firestore or aligning env with first-time user creation behavior.
- The ‚Äúprimary‚Äù administrator label in UI uses `getPrimaryAdminDisplayName()` from `src/config/adminEnv.ts` (defaults to ‚ÄúMIKI‚Äù if `VITE_ADMIN_DISPLAY_NAME` is unset). `isPrimaryAdminAccount` is true when the signed-in email equals the first entry in `VITE_ADMIN_EMAILS`.

3) What admin can do (this build)
----------------------------------
- Open `/admin` when authenticated with `role === 'admin'` (wrapped in `RequireRole allowed={['admin']}` in `src/App.tsx`).
- See the admin dashboard (`AdminPage`): session summary, permission model, honest placeholders for future moderation/analytics, and explicit ‚Äúnot implemented‚Äù sections.
- Use the same app features as a normal user (upload, profile, etc.) where Firestore rules allow ‚Äî admin is not a separate product login, only elevated routing/UX.

4) What admin cannot do yet
---------------------------
- Live user directory, reports queue, global video takedown from the admin screen, and platform-wide category CRUD are listed as placeholders only; there are no working controls that perform those actions.
- Dashboard metric tiles show ‚Äú‚Äî‚Äù and state that live data requires backend queries and Rules.

5) What regular (authenticated) users can do
---------------------------------------------
- Sign in, manage profile, use dashboard and upload (subject to `RequireAuth` and Firebase Rules), and use features wired to Firestore like comments/favorites/history where implemented.
- They cannot pass `RequireRole` for `/admin`; they get the ‚ÄúAccess denied‚Äù screen if they navigate there directly.

6) What guests can do
----------------------
- Browse public routes (home, watch, about, legal, `/admin-access` informational page) without signing in.
- Cannot access `/upload`, `/profile`, `/dashboard`, `/settings`, or `/admin` without authentication (and cannot access `/admin` without admin role).

7) What is protected in the UI only
-----------------------------------
- React route guards in `src/routes/guards.tsx` (`RequireAuth`, `RequireRole`) only control what the SPA renders. Hiding links in `Header` for non-admins is also UI-only.
- The admin email list in `VITE_ADMIN_EMAILS` is compile-time client configuration (discoverable in shipped JS). It must not be mistaken for server-side secrecy.

8) What still requires backend / Firebase Rules enforcement
-----------------------------------------------------------
- Any privileged read/write (moderation, user bans, bulk deletes, admin-only collections) must be denied by default in Firebase Security Rules and/or enforced by Cloud Functions or a trusted backend.
- Client-side role checks are explicitly documented on the access-denied view and in `AdminPage` as insufficient for production security.

9) Demo readiness
-----------------
- Suitable for demos: clear staff entry (`/admin-access`), admin dashboard with transparent boundaries, and a professional access-denied state for non-admin users on `/admin`.
- Ensure `.env` lists the demo admin email(s) and Firebase is configured so a real test account can sign in.

10) Production safety
---------------------
- Not sufficient for production security on its own: attackers can bypass browser routing. Production requires Rules (and/or server validation) for every sensitive operation and data path.
- Rotate and protect Firebase credentials; review `VITE_ADMIN_EMAILS` exposure and prefer Firestore-stored roles with Rules that validate `request.auth.token` or custom claims if you move to a stricter model.
