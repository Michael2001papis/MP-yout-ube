Purpose: Confirm why the app disables Firebase-backed features, validate that behavior matches code,
and list exact steps to clear the blocker. Generated from inspection of `src/services/firebase.ts`,
`.env.example`, `AuthPage`, `AdminEntryPage`, and Vite‚Äôs env model.

--------------------------------------------------------------------------------
REQUIRED VARIABLE NAMES (exactly six)
--------------------------------------------------------------------------------
As enforced in `src/services/firebase.ts`, every value must be a non-empty string after Vite resolves
`import.meta.env`. If any one is missing or empty, `firebaseReady === false` and `missingFirebaseEnv`
lists the keys that failed.

1. VITE_FIREBASE_API_KEY
2. VITE_FIREBASE_AUTH_DOMAIN
3. VITE_FIREBASE_PROJECT_ID
4. VITE_FIREBASE_STORAGE_BUCKET
5. VITE_FIREBASE_MESSAGING_SENDER_ID
6. VITE_FIREBASE_APP_ID

Optional for admin *labeling / role hints* only (do not satisfy Firebase init):
- VITE_ADMIN_EMAILS
- VITE_ADMIN_DISPLAY_NAME

--------------------------------------------------------------------------------
DOES .env.example MATCH RUNTIME EXPECTATIONS?
--------------------------------------------------------------------------------
Yes. `.env.example` documents the same six `VITE_FIREBASE_*` keys, states that all six are required,
notes Vite load timing (dev server start / build), local restart, and Vercel env + redeploy. Placeholder
values must be replaced with real Web app config from Firebase Console (Project settings ‚Üí Your apps).

--------------------------------------------------------------------------------
IS THE CURRENT CODE CORRECT?
--------------------------------------------------------------------------------
Yes, for the intended model:
- `getEnv` returns `''` for undefined/non-string; empty string counts as missing.
- `initializeApp` runs only when `missing.length === 0`.
- `auth`, `db`, `storage` are only created when `firebaseReady`.
- Services and `AuthContext` gate on `firebaseReady` / `assertFirebaseConfigured()`.

--------------------------------------------------------------------------------
IS THE BLOCKER CONFIG-ONLY OR CODE-RELATED?
--------------------------------------------------------------------------------
**Config-only.** The blocker appears when the six variables are absent or empty at the time Vite builds
or starts the dev server. No application bug is required for the symptom. Fixing values + restart/rebuild
resolves it.

--------------------------------------------------------------------------------
FAIL-CLOSED BEHAVIOR (VERIFIED)
--------------------------------------------------------------------------------
When `firebaseReady` is false:
- No Firebase app initialization; no real Auth/Firestore/Storage instances.
- `AuthContext` skips `onAuthStateChanged` and leaves user as guest (`AuthContext.tsx`).
- Auth UI: warning + `missingFirebaseEnv` list; Google and submit controls disabled (`AuthPage`).
- Admin entry: amber notice + missing keys list (`AdminEntryPage`).
- `loginWith*` / `logout` throw if called without config (`assertFirebaseConfigured`).
- Data services return safe defaults or no-ops (e.g. demo videos, empty arrays) per each service.

This is correct fail-closed behavior for a client-only Firebase web app.

LOCAL DEVELOPMENT: RESTART AFTER .env / .env.local CHANGES
--------------------------------------------------------------------------------
**Yes, required.** Vite embeds `VITE_*` at dev server startup and at production build time. Editing
`.env` or `.env.local` without restarting `npm run dev` typically leaves stale values in `import.meta.env`.

--------------------------------------------------------------------------------
VERCEL: SAME NAMES + FRESH DEPLOY
--------------------------------------------------------------------------------
**Yes.** Add the six variables in Vercel Project ‚Üí Settings ‚Üí Environment Variables (Production and any
Preview environments that build the app). Names must match exactly, including the `VITE_` prefix.

**Redeploy after adding or changing variables** so `vite build` runs with the new values. Old bundles
do not pick up new env at runtime for these keys.
