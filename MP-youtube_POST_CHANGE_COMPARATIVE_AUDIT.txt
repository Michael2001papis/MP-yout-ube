Nature of changes:

5. BOOT AND STARTUP REVIEW
================================================================================

App startup flow:
1. `main.tsx` renders `App` inside `BrowserRouter` and `StrictMode`.
2. `ThemeProvider` applies `light`/`dark`/`warm` classes on `document.documentElement`.
3. `AuthProvider` effect:
   - If `!firebaseReady`: sets `loading=false`, `user=null`, `role=guest`, **no global error**
     (explicit design — `src/context/AuthContext.tsx` lines 144–152).
   - If `firebaseReady`: subscribes `onAuthStateChanged`, loads/creates `users/{uid}` via `ensureUserDoc`.

Layout behavior:
- `Layout` shows Header always.
- Error banner only if `error && firebaseReady` (so missing Firebase does not spam red banner).
- If `loading`, **entire `<Outlet />` replaced** by “Loading session...” — blocks all routed pages until
  auth resolves when Firebase is configured.

Console / runtime:
- `firebase.ts` logs `console.warn` with missing keys when not ready — visible in devtools (acceptable
  for dev; slightly noisy on production builds if misconfigured).

Verdict:
- Startup is **predictable**; **unchanged** core tradeoff: global loading gate when Firebase is on.
- **Improved** operator clarity via `.env.example` and admin page messaging; not “cleaner” in console if
  env missing.
